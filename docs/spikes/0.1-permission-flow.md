# Spike 0.1 — Permission flow validation

Status: complete (against reference plugin source; live JSON-RPC trace not captured)
Date: 2026-05-17
claude version tested: 2.1.143 (binary at /Applications/cmux.app/Contents/Resources/bin/claude is a wrapper shell script around the bundled `cmux` CLI; `strings` extraction not viable)

Source of truth: aerolalit's official Telegram plugin at
`/Users/terrenceyodaiken/.claude/plugins/marketplaces/claude-plugins-official/external_plugins/telegram/server.ts`
(1038 lines — Nibbler's `:1402` citation is stale; current line numbers are :394 capability, :420 inbound handler, :773 / :930 outbound emit).

## What I confirmed

1. **Capability declaration requires BOTH keys** (server.ts:385–395):
   ```ts
   capabilities: {
     tools: {},
     experimental: {
       'claude/channel': {},            // base channel capability
       'claude/channel/permission': {}, // permission-relay opt-in
     },
   }
   ```
   Comment at :390 confirms semantic: "Declaring this asserts we authenticate the replier."

2. **Inbound notification** `notifications/claude/channel/permission_request` (CC → plugin),
   params schema (zod-validated at :420):
   `{ request_id: string, tool_name: string, description: string, input_preview: string }`.
   Matches Nibbler's hypothesis exactly.

3. **Outbound notification** `notifications/claude/channel/permission` (plugin → CC) at :772 and :929:
   ```ts
   { method: 'notifications/claude/channel/permission',
     params: { request_id, behavior: 'allow' | 'deny' } }
   ```

4. **request_id format**: 5 chars from `[a-km-z]` (lowercase a–z with `l` excluded —
   only the letter `l` is missing from the alphabet; `n` and `o` are valid).
   Regex at server.ts:84 and :733 is authoritative. Lowercase-only.

5. **No correlation ID beyond request_id**; plugin holds pending state in an in-memory `Map`
   (`pendingPermissions`, :412) keyed by request_id, used only for "See more" UI expansion.

## What I could not confirm (and why)

- **Live JSON-RPC byte trace**: `claude` binary is a shell wrapper; the real `cmux` CLI is
  separate, and no plugin scaffold exists in this worktree to run as MCP server. Building a
  minimal probe is in scope for Spike 0.2 (plugin scaffolding), not 0.1.
- **CC's behaviour when no permission capability is declared**: not tested. Spec §11.4 fallback
  logic assumes a non-permission path exists — likely the standard prompt-in-CC-host or
  `--permission-mode plan`. Reference plugin gives no hint because it always opts in.
- **Timeout/retry semantics**: reference plugin never times out a request; it relies on the
  user to click a button. If CC has a server-side timeout, we don't see it here.
- **`reason` field on deny**: NOT present in reference plugin. Either undocumented-but-supported,
  or Nibbler's spec invented it. Treat as unsupported until proven otherwise.

## Spec implications

- **§5.1 correction needed**: payload field is `behavior`, not `decision`. Drop `reason?` from
  the spec — it has no evidence. Add second required capability `claude/channel`.
- **§5.1 addition**: document request_id charset (`[a-km-z]{5}`) — useful for tests/fixtures.
- **§11.4 fallback**: untested by this spike. Recommend a follow-up micro-spike that boots a
  minimal MCP server WITHOUT `claude/channel/permission` and observes what CC does on a
  tool that would normally prompt. Until then, the fallback section is speculation.

## Recommendation

- **Sprint 4 branch**: **structured** — the channel-permission path is real, well-defined, and
  the reference implementation is small enough (~30 lines core) to mirror. No need to fall
  back to plan-mode-only.
- Block on Spike 0.2 (plugin scaffold) before attempting a live JSON-RPC capture; the
  reference plugin is sufficient evidence to start designing the bus contract now.
