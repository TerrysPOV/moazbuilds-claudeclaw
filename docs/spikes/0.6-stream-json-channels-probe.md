# Probe 0.6 — Can `-p --input-format=stream-json` replace PTY supervision?

Status: complete
Date: 2026-05-17
claude version: 2.1.143

## Q1: Multi-turn in one process?

Evidence: `printf '<json1>\n<json2>\n' | claude -p --input-format=stream-json --output-format=stream-json --verbose` returned two distinct `assistant`/`result` blocks for both turns under the same `session_id`. Driver script kept stdin open for ~30s and sent a second turn — claude processed it without restarting.

Verdict: **yes**. `--input-format=stream-json` is genuinely a long-lived multi-turn stream. Binary also exposes `--replay-user-messages` and `--sdk-url`, gated on `stream-json` both directions — strong confirmation this is the supported path.

## Q2: `--dangerously-load-development-channels` + `-p stream-json`?

Evidence: `claude -p --input-format=stream-json --output-format=stream-json --dangerously-load-development-channels probe-channel --mcp-config <file>` started cleanly, no flag-combination error. `init` event listed `{"name":"probe-channel","status":"connected"}` alongside other MCP servers. Binary strings show the gating logic (`approved channels allowlist`, `--dangerously-load-development-channels` for local dev) but no print-mode restriction.

Verdict: **flag combo is accepted; MCP server attaches**. No evidence of an explicit "channels disabled in `-p`" check.

## Q3: Channel notifications push into `-p` mode?

Evidence: Wrote a minimal stdio MCP server that advertises `experimental: { "claude/channel": {} }`, then sent `{"jsonrpc":"2.0","method":"notifications/claude/channel","params":{...,"message":"...CHIRP-42..."}}` 3s after init (claude was mid-generation, counting 1→20). Sent a second push at 13s. Asked claude in turn 2: "did you receive a CHIRP token?" — answer: **`NO_CHIRP`**. Session JSONL (`~/.claude/projects/.../<sid>.jsonl`) contains exactly 2 user turns (the ones I sent) and 2 assistant turns. No channel-injected user message, no error event, no rejection event in stream output.

Verdict: **no**. Channel push is silently dropped in `-p --input-format=stream-json`. Binary strings hint why: handler dispatches on `case "channel": return KI8(H,_.server,{midTurn:!0})` — channels arrive as "midTurn" interjections meant for the interactive REPL's input prompt, which doesn't exist in `-p`.

## Q4: JSONL written in this mode?

Evidence: `~/.claude/projects/-private-tmp-stream-json-probe/<sid>.jsonl` written for every probe run. Contains `user`, `assistant`, `attachment`, `last-prompt`, `queue-operation`, `system` event types — same schema as REPL sessions.

Verdict: **yes**. Bus's read path is unaffected.

## Q5: Slash commands via stdin?

Evidence: Sent `{"type":"user","message":{"role":"user","content":"/compact"}}` as a stream-json line. Output stream emitted `system/status: compacting` plus `SessionStart:compact` hook events. Process treated it as a real slash command, not text.

Verdict: **yes**, at least for `/compact`. (`/clear` and `/quit` not tested individually; high confidence given dispatcher is shared.)

## Bottom line

- Does this replace the bun-pty default? **NO**.
- Blocker: Q3. Channels (`notifications/claude/channel`) are the entire reason the Bus design picks claude up over a long-lived supervised process. Stream-json supervision gives us multi-turn + JSONL + slash commands but loses inbound channel pushes — the one capability Sprint 0 was protecting. Confirmed empirically with a working channel MCP server: notification sent, server connected, message never reached assistant context, no error surfaced.

## What this means for the spec

§5.3 stands. `bun-pty` remains the default supervisor because it preserves channel notifications end-to-end.

But this probe surfaces a strictly better **secondary** runner for the cases where channels aren't needed:

- **One-shots / cron jobs / non-channel triggers**: `process-stream-json` is a clean alternative — no PTY, no TTY emulation, cheaper to spawn, still gets JSONL persistence and slash commands. Consider exposing it as `supervisor: "stream-json"` in job config.
- **Don't demote bun-pty**. The channel-push delivery in REPL is the load-bearing capability for Telegram/Discord inbound, and stream-json silently drops it. Any switch would be a regression.
- **Action item for Sprint 1**: add a runner-selection branch — channel-bearing sessions use bun-pty, channel-free invocations use `process-stream-json`. Document this in §5.3 as a future optimisation, not a blocker.
- **Worth re-probing on every claude minor bump**: if Anthropic ever wires `notifications/claude/channel` into the stream-json event stream (e.g. as a new `{"type":"channel_message"}` event), PTY can be retired entirely. The plumbing is 90% there.
